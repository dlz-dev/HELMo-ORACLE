Architecture & Technologie

La Stack

Chaque composant a été choisi pour sa fiabilité, sa performance et sa compatibilité avec les exigences de souveraineté des données. Voici comment Oracle est construit.

Pipeline d'ingestion

Le pipeline d'ingestion transforme vos fichiers bruts en base de connaissances vectorielle interrogeable. Il est conçu pour être modulaire, robuste, et capable de traiter tous les formats documentaires courants.

Chaque fichier passe d'abord par un Gardien LLM qui vérifie que le contenu correspond au domaine thématique attendu. Les fichiers hors-sujet sont mis en quarantaine automatiquement. Les fichiers validés reçoivent ensuite une description contextuelle générée par LLM, préfixée à chaque fragment — ce qui améliore significativement la précision des recherches.

PDF · DOCX via Unstructured.io CSV · JSON · MD (parseurs natifs) Déduplication SHA-256 Late chunking contextuel Guardian LLM

Modèle d'embedding : nomic-embed-text · 768 dimensions · 8 192 tokens de contexte

Base de données vectorielle

Oracle stocke les embeddings dans PostgreSQL avec l'extension pgvector, hébergé directement sur le VPS dédié du client. Ce choix garantit l'isolation des données et la conformité RGPD — contrairement aux bases vectorielles cloud partagées.

Chaque document est stocké avec son contenu, son vecteur, ses métadonnées et son hash de déduplication. Pour les environnements de développement, ChromaDB est utilisé en local.

PostgreSQL + pgvector (production) ChromaDB (développement) Similarité cosinus Full-Text Search BM25 VPS dédié EU

Recherche hybride — Reciprocal Rank Fusion (RRF)

La recherche hybride est l'un des composants les plus critiques du pipeline RAG. Elle combine deux approches complémentaires : la recherche sémantique (qui comprend le sens d'une question) et la recherche par mots-clés BM25 (qui garantit la précision sur les termes rares, acronymes ou codes internes).

La fusion RRF fusionne les deux listes de résultats en attribuant un score à chaque document selon son rang dans chaque liste, plutôt que selon son score de similarité brut. Cette approche est plus robuste et favorise naturellement les documents pertinents pour les deux méthodes simultanément.

Recherche sémantique

Trouve les passages dont le sens est proche de la question, même avec des mots différents. Idéal pour les questions complexes et les synonymes.

Full-Text Search BM25

Trouve les passages contenant les termes exacts recherchés. Indispensable pour les codes d'erreur, numéros de référence, noms propres.

RRF — Reciprocal Rank Fusion Score : 1/(k + rang) k = 60 (constante standard)

Architecture agentique — MCP

Oracle implémente une boucle ReAct (Reasoning + Acting). Contrairement à un simple chatbot qui répond et s'arrête, l'agent Oracle peut enchaîner plusieurs raisonnements et actions : il analyse la demande, sélectionne l'outil MCP approprié, exécute l'action, observe le résultat, et itère jusqu'à satisfaire le besoin.

Le Model Context Protocol (MCP) standardise la communication entre le LLM et les outils externes. L'agent n'appelle pas les tools lui-même — il expose ses capacités et le client décide de les déclencher selon le contexte de la conversation.

LLM Judge — Évaluation automatique des réponses

Le LLM Judge est un composant d'observabilité qui s'exécute en arrière-plan après chaque réponse, de manière asynchrone et non-bloquante. Un modèle LLM dédié (température 0.0 pour maximiser le déterminisme) évalue la réponse sur 4 métriques standardisées.

L'architecture est fail-safe : si le juge échoue (timeout, JSON malformé, modèle indisponible), l'erreur est loggée silencieusement sans impacter l'expérience utilisateur. Les scores sont persistés en base de données et croisés avec les feedbacks manuels pour l'amélioration continue.

Sécurité & Conformité RGPD

La souveraineté des données est une contrainte de conception, pas une option. Chaque composant d'Oracle a été pensé pour garantir que les données sensibles restent dans le périmètre du client.

Le PII Masking (masquage des données personnelles) est actif à chaque tier, avec un niveau de sophistication croissant. En Tier 0–1, un filtre basique détecte les patterns courants (emails, téléphones, numéros de sécurité sociale). En Tier 2 Sovereign, le filtrage est renforcé. En Tier 3 Ultra, des modèles NER (Named Entity Recognition) lourds sont utilisés pour une détection quasi-exhaustive.

Tier 0–1 · Basique

Détection par patterns regex. Emails, téléphones, numéros courants. Couverture ~85%.

Tier 2 · Renforcé

Patterns enrichis + validation contextuelle. Noms, adresses, données bancaires. Couverture ~95%.

Tier 3 · Chirurgical

Modèles NER lourds entraînés spécifiquement. Entités nommées complexes. Couverture >99%.

Modèles de langage

Oracle est agnostique vis-à-vis du fournisseur LLM. La configuration du modèle est centralisée et peut être modifiée sans redéploiement. L'architecture sépare clairement le modèle de conversation, le modèle d'embedding, et le modèle juge.

Pour la conversation, les modèles légers et rapides (Flash / Haiku) sont utilisés par défaut pour minimiser la latence et les coûts. Les tiers supérieurs peuvent configurer des modèles plus puissants selon leur besoin de précision. Le juge utilise systématiquement un modèle à température 0.0 pour garantir le déterminisme des évaluations.

Claude Haiku (Anthropic) Gemini Flash (Google) GPT-4o mini (OpenAI) nomic-embed-text (embeddings) Ollama (local) Groq (juge — faible latence)
